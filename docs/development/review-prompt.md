## レビュー観点

### 1. Phase1、Phase2の内容との整合性

- Phase1で定義された要件と、Phase3の計画が整合しているか
- Phase2で設計されたシステム構成（React + TypeScript + Vite、LocalStorage永続化、TimerPageを画面調停レイヤーとする、usePomodoroTimerとuseWorkHistoryの分離、タイマー完了時にWorkSessionを保存、GitHub ActionsによるCI/CD、AWS S3/CloudFrontデプロイ）が計画に反映されているか

### 2. Phase3としての実装に踏み込みすぎていないか

- 開発計画の内容が、実装フェーズ（Phase4以降）への準備段階であることを確認
- 現在の計画は、具体的なコード実装を含まないことが適切か

### 3. 実装に踏み込みすぎている箇所のチェック

- Phase2で設計済みのコンポーネント名、Hook名、Service名、型名をタスク名として使うことは許容する
- ただし、実装コード、関数内部の詳細ロジック、コードサンプルには踏み込んでいないか確認する
- ファイルパスは成果物や設計済み配置先の参照として使うことは許容する
- ただし、コードレベルの実装指示になっていないか確認する

### 4. Git運用の実務的かどうか

- Gitブランチ戦略（main、develop、feature/*）が実務的な運用に適っているか
- Pull Requestの作成手順やマージルールが明確で実施可能か
- コミットメッセージの形式が適切か

### 5. Issue / PR単位が大きすぎないか

- 各Issue・PRの粒度が適切に分割されているか
- 1つのPRに複数の機能を含まないように設計されているか

### 6. CI/CDとAWSリリースまで見通せているか

- GitHub ActionsによるCI/CDの設定が計画されているか
- AWS S3/CloudFrontへのデプロイ手順が明確か
- デプロイ自動化の構成が実施可能か

## レビュー結果確認事項

- [ ] Phase1・Phase2の要件と設計との整合性
- [ ] 開発計画として適切な内容かどうか
- [ ] 実装に踏み込みすぎていないか
- [ ] Gitワークフローが実務的か
- [ ] Issue/PR単位の粒度が適切か
- [ ] CI/CDとAWSリリースの計画が明確か

## レビュー結果の記録方法

レビュー後、以下の点を確認し記録してください：

1. すべてのレビュー観点について確認済みであること
2. 計画内容に問題がないことを確認
3. 必要であれば修正提案を行うこと
4. レビュー結果をコメントとして記録すること

## レビュー完了後のアクション

- Phase3計画が承認された場合、次はPhase4（環境構築）の実装フェーズに進む
- 承認されなかった場合は、必要に応じて計画の見直しが行われる

## 修正確認事項

以下の変更点を確認してください：

1. development-plan.md の「実装対象外スコープ」が適切に定義されているか
2. task-breakdown.md でタスクが各フェーズに適切に分類されているか
3. 各タスクの完了条件が明確かつ実施可能か
4. Gitワークフローのブランチ戦略とルールが実務的かつ一貫性があるか
5. 各Issue/PRのタイトルが適切な粒度で、内容を正確に反映しているか
6. CI/CDとAWSリリースの計画が具体的かつ実施可能か
