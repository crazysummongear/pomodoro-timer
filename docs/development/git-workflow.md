# Pomodoro Timer Gitワークフロー

## 1. ブランチ戦略

### main
- リリース用の安定版コードを保持するブランチ
- マージはプルリクエスト経由のみとする
- CI/CDパイプラインで自動デプロイされる

### develop
- 開発用のメインブランチ
- 各機能開発はこのブランチから切り出し、マージされる
- テストが通る状態を維持する

### feature/*
- 各機能単位の開発用ブランチ
- ブランチ名は `feature/issue-number-description` の形式とする
- 例: `feature/1-setup-vite-project`, `feature/2-configure-eslint-prettier`

## 2. 各ブランチの役割

### main
- リリースされた安定版コードを保持
- テストが通っている状態を維持
- 本番環境へのデプロイ先

### develop
- 開発中の最新コードを保持
- 機能追加や修正のベースブランチ
- CI/CDパイプラインでテスト実行

### feature/*
- 新機能や修正の開発を行うための作業ブランチ
- レビュー後、developブランチにマージされる
- マージ後は削除する

## 3. ブランチ作成ルール

1. 開発開始前にdevelopブランチから新規ブランチを切り出す
2. ブランチ名は `feature/issue-number-description` の形式とする
3. 各作業は1つのIssue単位で行う
4. ブランチの作成後、対応するIssueをプルリクエストにリンクする

## 4. コミットメッセージルール

### 形式
```
<type>(<scope>): <subject>

<body>

<footer>
```

### type
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードフォーマット、セミコロンなど無関係な変更
- `refactor`: 機能変更なしのコード変更
- `test`: テスト追加・修正
- `chore`: ビルド処理、ツール変更など

### scope
- コンポーネント名やファイル名を記述
- 例: `feat(timer)`, `fix(utility)`, `test(hook)`

### subject
- 1行で要約を記述
- 動詞で始める（過去形ではなく、命令形）
- 例: "add timer display component", "fix session storage bug"

## 5. Pull Requestルール

### PR作成手順
1. 開発ブランチからdevelopブランチへのマージ用PRを作成
2. PRタイトルはIssue番号と要約で記述
   - 例: "Fix #3: Timer controls not working"
3. PRの説明には以下の情報を含める
   - 変更内容の概要
   - Issue番号のリンク
   - 動作確認手順
4. レビュアーを指定し、レビュー依頼を行う

### PRマージルール
- レビューが完了するまでマージしない
- CIテストが通っていることを確認
- コミットメッセージが適切か確認
- 作業ブランチはマージ後に削除する

## 6. レビュールール

### レビュアーの役割
1. コードの品質を確認
2. テストカバレッジを確認
3. 型定義やコンポーネント設計の妥当性を確認
4. レビュー用コメントを追加

### レビュー基準
- コードが意図通りに動作すること
- テストが網羅的で適切であること
- コメントが適切であること
- パフォーマンスやセキュリティに関する問題がないこと

## 7. マージルール

1. PRがレビュー完了し、承認されるまでマージしない
2. CIテストがすべて通過することを確認
3. 開発ブランチの変更内容をdevelopブランチにマージ
4. 必要に応じて再作成・再確認を行う

## 8. 禁止事項

- mainブランチへの直接コミットは禁止
- developブランチへの直接プッシュは禁止
- テストが通っていないコードをマージしない
- コミットメッセージが不適切な場合は修正後マージ
- ブランチの削除後にマージした場合、再開発が必要になるため禁止

## 9. 今回の学習プロジェクトでの運用例

### 開発フロー例
1. developブランチから `feature/1-timer-component` ブランチを切る
2. タイマー画面コンポーネントの実装を行う
3. ローカルでテストし、コミットする
4. PRを作成し、レビューを受ける
5. レビューが完了したらdevelopブランチにマージ
6. featureブランチを削除

### テストフロー例
1. 開発ブランチのコードをdevelopブランチにマージ
2. CI/CDパイプラインでテスト実行
3. テストが通らない場合は修正し、再度マージ

### リリースフロー例
1. developブランチの安定版をmainブランチにマージ
2. バージョン管理を行い、GitHub Releaseを作成
3. CI/CDパイプラインで本番環境へデプロイ

### 開発フロー例（Phase4以降）
1. developブランチから `feature/4-setup-project` ブランチを切る
2. Viteプロジェクトの初期設定を行う
3. ローカルでテストし、コミットする
4. PRを作成し、レビューを受ける
5. レビューが完了したらdevelopブランチにマージ
6. featureブランチを削除

### テストフロー例（Phase6以降）
1. 開発ブランチのコードをdevelopブランチにマージ
2. CI/CDパイプラインでテスト実行
3. テストが通らない場合は修正し、再度マージ

### リリースフロー例（Phase7以降）
1. developブランチの安定版をmainブランチにマージ
2. バージョン管理を行い、GitHub Releaseを作成
3. CI/CDパイプラインで本番環境へデプロイ
