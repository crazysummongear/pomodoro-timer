# Cline 作業依頼テンプレート

## 1. 目的

本書は、CodexがClineへ作業を依頼する際の標準テンプレートを定義する。

目的は以下である。

- Clineへの指示粒度を安定させる
- 工程逸脱を防ぐ
- 読み込むドキュメントを最小化する
- 変更対象ファイルを明確にする
- 完了報告と実ファイルの不一致を防ぐ

## 2. 基本方針

Clineへの依頼は、必ず以下の順序で行う。

1. 作業前提を伝える
2. 今回の工程を明示する
3. 読むファイルを限定する
4. 変更してよいファイルを限定する
5. 禁止事項を明示する
6. 作業内容を小さなステップに分ける
7. 完了前チェックを指定する
8. 完了報告形式を指定する
9. レビュー待ちで停止させる

## 3. Clineへ必ず最初に読ませる文書

```md
作業前に必ず以下を読んでください。

- docs/process/cline-working-guide.md

読んだうえで、今回依頼された対象ファイルだけを変更してください。
完了報告前に、実ファイルを読み直して修正内容が反映されていることを確認してください。
```

## 4. 標準依頼テンプレート

```md
あなたはPomodoro Timerプロジェクトの作業担当です。
作業前に必ず `docs/process/cline-working-guide.md` を読んでください。

# 今回の工程

<例: 工程監査是正 / Phase4 環境構築 / Phase5 実装>

# 今回の目的

<今回達成したいことを1〜3行で記載>

# 今回読むファイル

以下のファイルだけ読んでください。

- <file path>
- <file path>

大量のドキュメントを一度に読まないでください。
必要になった場合は、追加で読むべきファイルを報告してください。

# 変更してよいファイル

今回は以下のファイルのみ変更してください。

- <file path>
- <file path>

# 変更してはいけないもの

- 指定外のファイル
- 実装コード
- package.json
- npm installを伴う変更
- Git操作
- GitHub Actions
- AWS設定
- 次工程の成果物

# 作業内容

以下の順に対応してください。

1. <Step 1>
2. <Step 2>
3. <Step 3>

# 完了前チェック

完了報告前に、必ず実ファイルを読み直して以下を確認してください。

- 指摘事項が実ファイルに反映されている
- 古い文言が残っていない
- 章番号が重複していない
- 工程状態に矛盾がない
- 指定外ファイルを変更していない
- 次工程へ進んでいない

# 完了報告

以下の形式で報告してください。

## 作業結果

- 修正したファイル:
- 修正したセクション:
- 実施した内容:

## セルフレビュー

- 実ファイル確認済み:
- 指摘事項への対応:
- 変更していないファイル:
- 工程逸脱なし:

## 残課題

- なし / あり

## 次工程

- レビュー待ち
- 承認されるまで次工程へ進まない
```

## 5. レビュー指摘修正用テンプレート

```md
レビュー結果を反映してください。
作業前に必ず `docs/process/cline-working-guide.md` を読んでください。

今回はレビュー指摘の反映のみを行います。
実装、環境構築、npm install、Git操作、GitHub Actions作成、AWS設定は行わないでください。

# 対象ファイル

- <file path>

# 指摘事項

## 1. <指摘タイトル>

現在:

- <現在の問題>

修正方針:

- <修正後の内容>

## 2. <指摘タイトル>

現在:

- <現在の問題>

修正方針:

- <修正後の内容>

# 完了前チェック

- 対象ファイルを読み直した
- 指摘事項がすべて反映されている
- 古い文言が残っていない
- 指定外ファイルを変更していない

# 完了報告

修正後、以下を報告してください。

- 修正したファイル
- 修正したセクション
- 各指摘への対応状況
- 実ファイル確認済みであること
- レビュー待ちで停止していること
```

## 6. 新規成果物作成用テンプレート

```md
新規成果物を作成してください。
作業前に必ず `docs/process/cline-working-guide.md` を読んでください。

# 今回の工程

<工程名>

# 作成するファイル

- <file path>

# まず読むファイル

以下だけ読んでください。

- <file path>
- <file path>

# 作成内容

以下の章立てで作成してください。

1. 文書管理
2. 目的
3. 対象範囲
4. 内容
5. レビュー観点
6. 完了条件

# 禁止事項

- 指定外ファイルを変更しない
- 実装しない
- 次工程へ進まない
- 要件外の内容を追加しない

# 完了報告

- 作成したファイル
- 各章の概要
- 既存成果物との整合性
- セルフレビュー結果
- レビュー待ちで停止していること
```

## 7. Phase4 環境構築依頼テンプレート

Phase4へ進む場合は、以下のように範囲を明確にする。

```md
あなたはPomodoro Timerプロジェクトの作業担当です。
作業前に必ず `docs/process/cline-working-guide.md` を読んでください。

# 今回の工程

Phase4 環境構築

# 今回の目的

React + TypeScript + Vite の開発環境を構築する。
ただし、アプリケーション機能の実装は行わない。

# 読むファイル

- docs/process/phase-gate-status.md
- docs/development/development-plan.md
- docs/development/task-breakdown.md
- docs/development/git-workflow.md

# 作業範囲

実施してよい作業:

- Viteプロジェクト初期化
- ESLint / Prettier設定
- Vitest設定
- 基本ディレクトリ作成
- 開発サーバー起動確認
- 初期テスト実行確認

実施してはいけない作業:

- タイマー機能の実装
- LocalStorage連携の実装
- UIコンポーネント実装
- GitHub Actions作成
- AWS設定
- デプロイ

# 完了条件

- package.json が作成されている
- React + TypeScript + Vite が起動できる
- ESLint / Prettier / Vitest が設定されている
- src配下の基本ディレクトリが作成されている
- 機能実装を行っていない

# 完了報告

- 作成・変更したファイル
- 実行したコマンド
- 確認した動作
- 実装に踏み込んでいないこと
- Phase5へは進まずレビュー待ちで停止していること
```

## 8. Codexレビュー時の確認ポイント

Clineの作業後、Codexは以下を確認する。

- 指定ファイルだけ変更されているか
- 工程状態が正しいか
- 報告内容と実ファイルが一致しているか
- 不要な実装や設定が混入していないか
- 次工程へ勝手に進んでいないか
- Git差分がレビュー可能な粒度か

## 9. 運用ルール

- Clineへの依頼は1回につき1目的にする。
- 大きなタスクは複数プロンプトに分割する。
- Clineの完了報告だけで承認しない。
- Codexが必ず実ファイルを確認してレビューする。
- 承認後にGitへコミットし、GitHubへ反映する。
