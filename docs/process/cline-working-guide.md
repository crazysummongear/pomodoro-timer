# Cline 作業ガイド

## 1. 目的

本書は、ClineがPomodoro Timerプロジェクトで作業する前に必ず確認する作業ルールを定義する。

目的は以下である。

- 工程順序を守る
- 承認前に次工程へ進まない
- 実装範囲を勝手に広げない
- 報告内容と実ファイルの不一致を防ぐ
- ローカルLLMでも迷わないように、読む範囲と作業範囲を小さく保つ

## 2. Clineの役割

Clineは主に以下を担当する。

- 指定された工程の成果物作成
- 指定されたファイルの修正
- セルフレビュー
- 修正内容の報告

Clineは以下を勝手に行ってはならない。

- 次工程への進行
- 実装開始
- 環境構築
- npm install
- Git操作
- GitHub Actions作成
- AWS設定
- 要件外機能の追加
- 既存方針と矛盾する判断

ただし、ユーザーまたはレビュワーが明示的に依頼した場合は、その依頼範囲に限って作業してよい。

## 3. 工程順序

本プロジェクトは以下の順序で進める。

1. Phase1 要件定義
2. Phase2 システム設計
3. Phase3 開発計画
4. 工程監査是正
5. Phase4 環境構築
6. Phase5 実装
7. Phase6 テスト
8. Phase7 AWSリリース
9. Phase8 運用改善

前工程の成果物レビューと承認が完了するまで、次工程へ進んではならない。

## 4. 作業前に必ず確認すること

作業前に以下を確認する。

- 現在の工程
- 今回依頼された作業範囲
- 変更してよいファイル
- 変更してはいけないファイル
- 実装を伴う作業か、ドキュメントのみか
- 承認待ちで停止すべきか

不明な場合は、推測で進めず確認事項として報告する。

## 5. 読み込みルール

ローカルLLMであることを考慮し、一度に大量のドキュメントを読まない。

原則として、以下の順で必要最小限だけ読む。

### 5.1 工程確認時

最初に読むファイル:

- `docs/process/phase-gate-status.md`
- `docs/process/process-audit.md`

確認すること:

- 現在工程
- 各Phaseの状態
- Phase4以降へ進める条件
- 不足成果物

### 5.2 要件確認時

読むファイル:

- `docs/requirements/requirements.md`

確認すること:

- スコープ
- 対象外
- 機能要件
- 非機能要件
- 受け入れ条件

### 5.3 設計確認時

必要に応じて、以下を1つずつ読む。

- `docs/architecture/architecture.md`
- `docs/architecture/sequence-diagram.md`
- `docs/design/screen-design.md`
- `docs/design/component-design.md`
- `docs/architecture/adr/0001-react-adoption.md`
- `docs/architecture/adr/0002-s3-cloudfront.md`

### 5.4 開発計画確認時

読むファイル:

- `docs/development/development-plan.md`
- `docs/development/task-breakdown.md`
- `docs/development/git-workflow.md`

## 6. 作業範囲ルール

依頼された対象ファイルのみ変更する。

例:

- `docs/process/process-audit.md` の修正依頼であれば、そのファイルだけを変更する。
- 複数ファイルが指定された場合は、指定されたファイルだけを変更する。
- 指定外のファイルに気になる点があっても、勝手に修正しない。

必要な修正が指定外ファイルにある場合は、完了報告の「追加提案」に書く。

## 7. 実装禁止ルール

以下の工程では実装を行わない。

- Phase1 要件定義
- Phase2 システム設計
- Phase3 開発計画
- 工程監査是正

これらの工程で禁止される作業:

- `src/` 配下のコード作成
- `package.json` 作成
- npm install
- Vite初期化
- ESLint / Prettier / Vitest設定ファイル作成
- GitHub Actions YAML作成
- AWS関連設定

Phase4以降でも、依頼された工程範囲を超えてはならない。

## 8. 報告前チェック

完了報告前に、必ず実ファイルを読み直して確認する。

確認項目:

- 指摘された箇所が実際に修正されている
- 古い文言が残っていない
- 章番号が重複していない
- Phase状態に矛盾がない
- 「未着手」「進行中」「完了」「承認済み」の表現が正しい
- 依頼されていないファイルを変更していない

報告内容は、実ファイルの内容と一致させる。

## 9. 工程状態の表現ルール

工程状態は以下の意味で使う。

| 状態 | 意味 |
| --- | --- |
| 未着手 | 作業を開始していない |
| レビュー用ドラフト | 成果物は作成済みだがレビュー前またはレビュー中 |
| レビュー待ち | レビュー依頼済みで承認待ち |
| 実質承認済み | 会話上は承認済みだが、文書上の承認記録が未整備 |
| 承認済み | 文書上も承認状態が記録されている |
| 完了 | 成果物、レビュー、承認がすべて完了している |

Phase4は、環境構築を開始するまで `未着手` とする。

## 10. スコープ逸脱防止

以下は初期リリースの対象外である。

- ユーザー認証
- バックエンドAPI
- データベース
- 複数端末同期
- AI記事レビュー
- Note記事管理
- ネイティブアプリ
- 通知機能
- カスタムテーマ

対象外機能を設計・実装してはならない。

## 11. よくあるミス

### 11.1 報告だけ修正済みになっている

実ファイルが直っていない状態で「修正済み」と報告してはならない。

対策:

- 完了報告前に対象ファイルを読み直す。
- 修正後の該当見出しや表の内容を確認する。

### 11.2 Phase4を進行中にしてしまう

環境構築を実施していない場合、Phase4は `未着手` である。

### 11.3 実装未着手なのにトレーサビリティをOKにする

実装やテストが未着手の場合は `未評価 / 未着手` とする。

### 11.4 AI利用を「行われていない」と書く

本プロジェクトはAI駆動開発である。

正しい表現:

- AIによるドキュメント作成支援は行われている。
- ただし、レビューと承認ゲートにより勝手な機能追加を防止している。

## 12. 完了報告テンプレート

作業完了時は以下の形式で報告する。

```md
## 作業結果

- 修正したファイル:
- 修正したセクション:
- 実施した内容:

## セルフレビュー

- 実ファイル確認済み:
- 指摘事項への対応:
- 変更していないファイル:
- 工程逸脱なし:

## 残課題

- なし / あり

## 次工程

- レビュー待ち
- 承認されるまで次工程へ進まない
```

## 13. Clineが迷った場合の判断基準

迷った場合は、以下を優先する。

1. 工程順守
2. 既存ドキュメントとの整合性
3. スコープを広げない
4. 変更範囲を小さくする
5. 推測で進めず、確認事項として報告する

## 14. レビュワーへの確認事項

作業後、以下をレビュワーに確認してもらう。

- 工程順序が守られているか
- 成果物の内容に矛盾がないか
- 指摘事項が実ファイルに反映されているか
- 次工程へ進める状態か
- 追加の是正が必要か
