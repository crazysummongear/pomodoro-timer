## 1. Phase1の目的

Phase1では、設計や実装に入る前に、Pomodoro Timer の業務要件・機能要件・非機能要件を明確にする。

本プロジェクトの主目的は、ポモドーロタイマーそのものを完成させることではなく、実務に近い開発プロセスを経験することである。

このフェーズでは、以下を関係者間で合意する。

- なぜこのシステムを作るのか
- 誰が利用するのか
- 初期リリースで何を作るのか
- 初期リリースで何を作らないのか
- どの品質要件を重視するのか
- 次フェーズであるシステム設計へ進むために何を確認するのか

## 2. システム概要

Pomodoro Timer は、個人利用を目的としたブラウザベースの生産性支援アプリケーションである。

利用者は作業用タイマーを開始・停止・リセットでき、作業履歴をブラウザの LocalStorage に保存できる。

初期リリースでは、機能を最小限に保ちつつ、以下を学習できる構成を目指す。

- 要件定義からリリースまでの工程管理
- GitHubを利用したチーム開発の模擬体験
- Pull Requestとレビューを前提とした開発
- GitHub ActionsによるCI/CD
- AWS S3 / CloudFrontへの静的Webアプリケーション配信
- 将来的なNote記事管理ツールやAI記事レビューアへ発展可能な構成

## 6. 機能要件

### FR-001 タイマー開始

- ユーザーはメイン画面からタイマーを開始できる。
- タイマー開始後、残り時間がカウントダウンされる。
- 画面上でタイマーが実行中であることが分かる。

### FR-002 タイマー停止

- ユーザーは実行中のタイマーを停止または一時停止できる。
- 停止中はカウントダウンが進まない。
- 停止した時点の残り時間から再開できる。

### FR-003 タイマーリセット

- ユーザーは現在のタイマーをリセットできる。
- リセット後、タイマーは初期時間に戻る。
- リセット操作によって、保存済みの作業履歴は削除されない。

### FR-004 作業履歴保存

- アプリケーションは作業セッションの履歴を記録できる。
- 履歴には最低限、以下の情報を含める。
  - セッション日時
  - 作業時間
  - 完了状態または記録状態
- ユーザーは保存済みの作業履歴を確認できる。

### FR-005 LocalStorage永続化

- 作業履歴はブラウザのLocalStorageに保存する。
- ページを再読み込みしても、保存済みの履歴を確認できる。
- LocalStorageのデータが存在しない場合や不正な形式の場合でも、アプリケーションが致命的に停止しない。

## 7. 非機能要件

| ID | 分類 | 要件 |
| --- | --- | --- |
| NFR-001 | 保守性 | React、TypeScriptを用い、責務が分かるディレクトリ構成にする。 |
| NFR-002 | テスト容易性 | タイマーの主要ロジックとLocalStorage操作はVitestでテスト可能にする。 |
| NFR-003 | 信頼性 | 通常操作でタイマー状態や作業履歴が破損しないようにする。 |
| NFR-004 | 性能 | 静的Webアプリケーションとして短時間で読み込める構成にする。 |
| NFR-005 | 利用性 | 主要操作である開始・停止・リセットが直感的に分かるUIにする。 |
| NFR-006 | 移植性 | モダンなPCブラウザおよびモバイルブラウザで利用できるようにする。 |
| NFR-007 | デプロイ容易性 | GitHub Actionsでビルドし、AWS S3 / CloudFrontへデプロイできる構成にする。 |
| NFR-008 | コード品質 | ESLintとPrettierにより、基本的な品質とフォーマットを維持する。 |
| NFR-009 | セキュリティ | リポジトリに秘密情報をコミットしない。AWS認証情報はGitHub ActionsのSecretsまたはOIDCで扱う。 |

## 11. リスク

| ID | リスク | 影響 | 対応方針 |
| --- | --- | --- | --- |
| R-001 | LocalStorageはユーザー操作やブラウザ設定により削除される可能性がある。 | 作業履歴が失われる。 | 初期リリースではローカル保存の制約として明記する。 |
| R-002 | 単純なinterval実装ではタイマーにずれが発生する可能性がある。 | 作業時間が不正確になる。 | Phase2でタイムスタンプを基準にした設計を検討する。 |
| R-003 | AWSデプロイ設定はアカウント固有の作業が必要になる。 | リリース作業が遅延する。 | リリース計画で前提条件と必要なSecretsを明確にする。 |
| R-004 | 機能を増やしすぎると、開発プロセス学習という目的から外れる。 | スコープが肥大化する。 | Phase1のスコープを小さく保ち、フェーズゲートを徹底する。 |
